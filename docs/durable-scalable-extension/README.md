# Lattice: Technical Design Document & Implementation Plan

This document provides a comprehensive and technical design for **Lattice**, an AI-powered 3D modeling platform. Lattice allows users to interact with a coding agent (`pi`) to generate 3D models (STLs) in real-time.

## 1. Project Overview

Lattice is a web application inspired by the architecture of `vibepdf` but adapted for 3D modeling.
*   **UI**: Two-panel layout. Left: 3D Preview (Three.js). Right: Chat.
*   **Agent**: `pi` coding agent (running in a sandboxed environment).
*   **CAD Engine**: Python with the `build123d` library.
*   **Persistence**: Supabase for sessions/auth (using the Blob/File model), S3 (or compatible) for files (STLs, code).
*   **Infrastructure**: Local Kubernetes (**k3s**) for isolated, ephemeral sandboxes.

## 2. Architecture & Tech Stack

*   **Frontend**: Next.js 15 (App Router), Tailwind CSS, `shadcn/ui`, `@react-three/fiber` (Three.js for 3D rendering).
*   **Backend**: Next.js API Routes (handling K8s orchestration and session management).
*   **Database & Auth**: Supabase (Postgres for session storage, GoTrue for Email/Password Auth).
*   **Storage**: Amazon S3 or Supabase Storage (for artifacts).
*   **Agent**: `@earendil-works/pi-coding-agent` (used in SDK or RPC mode).
*   **Sandbox**: Docker containers running in Kubernetes (**k3s**).

## 3. Session Management: The "Blob/File" Model

To ensure perfect compatibility with `pi`'s session format and avoid data corruption, we will use the **Blob/File Model** for session persistence.

### Database Schema
In Supabase, we will have a `sessions` table:
*   `id`: UUID (Primary Key).
*   `user_id`: UUID (References auth.users).
*   `name`: String (Project name).
*   `conversation_log`: **TEXT** (Stores the entire raw JSONL string generated by `pi`).
*   `created_at`: Timestamp.

### Load & Save Flow
1.  **Loading**: When a user opens a session, the backend fetches the `conversation_log` string from Supabase.
    *   **For the Frontend**: The backend parses the JSONL string on the fly (`string.split('\n').map(JSON.parse)`) and sends a clean JSON array of messages to the UI.
    *   **For the Agent**: The backend writes the raw string to a file (e.g., `/workspace/session.jsonl`) inside the Kubernetes Pod so `pi` can read it natively.
2.  **Saving**: When the session ends or periodically during interaction, the backend reads the JSONL file from the Pod and updates the `conversation_log` column in Supabase.

## 4. Sandbox Design (K3s + Docker)

Each user session corresponds to a logical "Sandbox" running in a local **k3s** cluster.

### Sandbox Dockerfile
The image running in the Pod will contain:
*   **Base**: `node:20` (required to run `pi` or the bridge service).
*   **Python Environment**: Python 3.10+, `pip`.
*   **CAD Libraries**: `build123d`, `cadquery` (installed via pip).
*   **Agent**: Pre-installed `@earendil-works/pi-coding-agent`.

### Kubernetes Orchestration
*   The Next.js backend will use the `@kubernetes/client-node` library.
*   **Lifecycle**:
    1.  User prompt received -> Check if a Pod exists for the session.
    2.  If not, create a Pod using a `Pod` or `Job` manifest.
    3.  Hydrate the Pod by downloading previous code files from S3 and the JSONL session from Supabase.
    4.  Execute the agent command.
    5.  Scale to Zero: If the Pod is idle for X minutes, terminate it to save local resources.

## 5. Real-time Communication (SSE)

To provide a live stream of the agent's thoughts and tool calls to the user:
*   **Protocol**: **Server-Sent Events (SSE)**.
*   **Flow**:
    1.  The agent in the Pod emits events (e.g., `tool_call: write file.py`).
    2.  The backend captures these events and forwards them as SSE events to the frontend.
    3.  The frontend appends these to the activity feed in real-time.
*   *Rationale*: SSE is simpler to implement than WebSockets for read-only streams and handles reconnection automatically.

## 6. Step-by-Step Implementation Plan

### Phase 1: Scaffolding & Cleanup (from VibePDF)
We will use `vibepdf` as a starter but remove unnecessary parts.
1.  **Copy**: Copy the Next.js project structure, Tailwind config, and basic UI shells.
2.  **Cleanup**:
    *   Remove `react-pdf` and the PDF viewer component.
    *   Remove `tesseract.js` (OCR) and `pdf-lib`.
    *   Delete API routes related to PDF manipulation (`/api/upload`, etc.).
3.  **Rename**: Update all branding, folder names, and variables from VibePDF to Lattice.

### Phase 2: Authentication & Database
1.  **Supabase Setup**: Create the project and configure **Email/Password** authentication.
2.  **Schema Application**: Create the `sessions` table with the `conversation_log` TEXT column.

### Phase 3: Local K3s & Sandbox Setup
1.  **Install K3s**: Set up a local K3s cluster. Add scripts to the repo to install/uninstall and flush the cluster.
2.  **Build Image**: Create the Sandbox Dockerfile with Python and `build123d`.
3.  **K8s Manifests**: Write the YAML templates for the Pods and NetworkPolicies (blocking egress).

### Phase 4: Agent Integration & SSE
1.  **Backend Bridge**: Write the API route that handles communication between Next.js and the K8s Pod.
2.  **SSE Setup**: Implement the SSE endpoint to stream `pi` events to the UI.
3.  **UI Updates**: Update the chat panel to display streaming tool calls.

### Phase 5: Artifact Persistence
1.  **Execution**: Agent runs code to generate the STL.
2.  **S3 Sync**: Upload the STL and the source code file to S3.
3.  **Preview**: Load the STL from the S3 signed URL into the Three.js canvas on the left panel.
